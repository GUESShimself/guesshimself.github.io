Building a Multi-Platform Design System

Leading the creation of a token-driven design system that unified experiences across web, iOS, and Android for a Fortune 500 financial services company.

Company
Capital One
Date
2021–2023
Role
Lead Designer
Scope
Design tokens, components, documentation
Team
4 designers, 6 engineers, 1 PM
Platform
Web, iOS, Android

Impact

Key Outcomes

  • Reduced design-to-dev handoff time by 60% through shared token vocabulary
  • Shipped 40+ production-ready components across three platforms
  • Achieved 85% adoption across 12 product teams within first year
  • Established governance model that balanced consistency with team autonomy
  • Created comprehensive documentation site with live code examples

Context & Constraints

The company had grown rapidly through acquisitions, resulting in fragmented experiences across products. Each team used different patterns, spacing scales, and color systems. Users encountered different button styles, form behaviors, and navigation patterns depending on which product they used.

Engineering teams maintained duplicated component libraries with inconsistent APIs. Designers created one-off solutions for common problems. Accessibility was inconsistent. The organization needed a unified system without disrupting ongoing product development.

Core Constraint

The system needed to serve three distinct platforms (web, iOS, Android) with different technical stacks, while maintaining visual and behavioral consistency. We couldn't force platform convergence—the system had to respect platform conventions while providing a cohesive brand experience.

Problem

  • Fragmented user experience: Inconsistent patterns across products eroded trust and increased cognitive load
  • Inefficient workflows: Teams rebuilt common components from scratch, duplicating effort
  • Accessibility gaps: No shared standards meant inconsistent keyboard navigation, screen reader support, and color contrast
  • Scaling challenges: Adding new features or updating brand guidelines required coordinating changes across dozens of teams
  • Technical debt: Multiple competing component libraries with incompatible APIs and dependencies

Approach

System Foundations

We started with design tokens—the atomic values that define the system's visual language. Rather than hardcoding colors or spacing, we created a semantic token structure that mapped brand values to platform-specific implementations. A single source of truth in JSON generated platform-specific code for web (CSS custom properties), iOS (Swift), and Android (XML).

Component design followed a mobile-first approach with responsive behaviors defined at the token level. We documented not just what components looked like, but when to use them, common pitfalls, and accessibility requirements.

Adoption & Enablement

We embedded with product teams during the first implementations, learning what worked and what created friction. This revealed that teams needed migration paths, not just greenfield guidance. We created codemods and design file migrations to ease transitions.

Office hours, Slack channels, and regular showcases built community. We treated early adopters as partners, incorporating their feedback into the system roadmap.

Governance

A lightweight contribution model let teams propose new components or variants. We established a review process that balanced consistency with flexibility—approving contributions that solved real needs while maintaining system coherence. Monthly design system office hours and quarterly roadmap reviews kept stakeholders aligned.

Solution

Results

  • Faster delivery: Teams using the system shipped features 40% faster than teams building custom components
  • Consistent quality: Accessibility audits showed 95% WCAG AA compliance across system components
  • Sustained adoption: After initial rollout, usage grew organically as teams experienced the benefits
  • Cultural shift: Design system thinking became part of the product development process, not an afterthought

The system evolved from an internal tool into a strategic asset. Product teams could focus on differentiated experiences rather than rebuilding foundations. The shared vocabulary improved cross-functional collaboration. When the company updated its brand, we rolled out changes across all products in weeks instead of months.

Reflection

What I'd Refine Further

The token structure worked well for visual properties but struggled with motion and interaction patterns. If I were starting over, I'd invest earlier in animation tokens and shared interaction behaviors. We also underestimated the effort required to keep documentation in sync with code—better automation here would have reduced maintenance burden.

What This Changed About My Thinking

This project reinforced that successful design systems are as much about people and process as they are about components and tokens. The technical architecture mattered, but what drove adoption was building trust through partnership, creating clear contribution paths, and demonstrating value through team enablement rather than top-down mandates.

I also learned that premature abstraction is expensive. Starting with concrete use cases and extracting patterns as they emerged was more effective than trying to anticipate every future need upfront. Design systems should be opinionated about foundations but flexible about applications.
